Chapter 10 - Debugging

The following notes are of particular areas that I was less familiar with, didn't fully realize, or would like to have as a quick reminder.

  • Exception: return value can be used by the program to deal with the error.
  • Stack Trace: works backwards from the point of the error to the original method that started the sequence.
  • Warning: program will continue to run but may not run correctly.
  • Runtime Error: HTML will still appear, but JS code will stop working in the background.
  • Ensure code fails gracefully (e.g. without exceptions thrown) but loudly in Development (using TDD).
  • Strict Mode: produces more exceptions and warnings and prohibits deprecated features. Add 'use strict'; to first line of JS file.
  • Recommended way to invoke strict mode is to place all of your code into a self-invoking function as in (function(){'use strict';}());
  • Modules use strict mode by default, so no need for 'use strict' declaration.
  • Linting Tools: JS Lint, JS Hint, ES Lint. Installation options include: editor plugins, onlline tools, installed on computer using npm.
  • Feature Detection: use an if statement to check whether an object or method exists before using it, such as if(window.holoDeck){//code here}
  • Browser sniffing is not recommended because user agent string cannot be relied upon to be accurate.
  • Debugging in the Browser
    • Alert
    • Console.log()
    • Console.trace()
    • Breakpoints in the browser
    • Debugger keyword: will create a breakpoint in your code that will pause the execution of the code and allow you to see where the program is currently up to. Can hover over variables. Can restart by clicking the 'play' button. Remember to remove debugger prior to deployment.
  • Error Objects
    • Can be created in the host environment.
    • Can be created in code using a constructor function as in const error = new Error();
    • Other Error Objects
      • EvalError
      • RangeError
      • ReferenceError
      • SyntaxError
      • TypeError
      • URIError
      • InternalError
      • Can be used as constructors to create custom error objects as in const error = new TypeError('You need to use numbers in this function');
      • Error objects have three "safe to use" properties: name, message, and stack.
    • Throwing Exceptions
      • You can throw your own exceptions using the throw statement
      • Causes the execution of the program to stop.
      • Best practice to throw an error object, as in throw new Error('Something has gone wrong');, which can then be caught in a catch block.
    • Exception Handling
      • Catching an error is a way to handle errors gracefully so that the user does not see the errors.
      • try...catch...finally
      • Code inside the catch block will only run in an exception is thrown inside the try block. Error object is automatically passed as a parameter to the catch block, which allows you to query the error name, message, and stack properties.
      • Code inside the finally block will always be executed after the try or catch block, regardless of whether an exception occured or not. Useful if you want code to run in both cases.
    • Tests
      • A test can simply be a function that tests a piece of code runs as it should. For example, you could create a function as in function itSquareRoots4(){return squareRoot(4)===2;}. In that example, we are comparing the result of squareRoot(4) with the number 2, which returns true if our function works as expected.
    • Test Driven Development (TDD)
      • Write Tests (that initially fail)
      • Write code to pass the tests
      • Refactor the code
      • Test refactored code
      • Write more tests for new features
